1 The Problem Feast Solves

Before Feast, ML teams suffered from a set of recurring, expensive problems. Understanding these problems is the key to understanding why Feast exists.

Problem 1 — Training/Serving Skew

This is the most dangerous problem in production ML. Your model trains on data computed one way, but at serving time the same features are computed differently. The model works in notebooks but fails silently in production.

🧑‍💻
Data Scientist
Computes features in pandas during training
⚙️
Engineer
Re-implements same features in Java/Go for serving
⚠️ Two implementations = silent divergence = model degrades in prod

Problem 2 — Feature Duplication Across Teams

Team A builds "customer lifetime value." Team B independently builds the same thing. No discoverability. No reuse. Every team reinvents the wheel.

Problem 3 — Data Leakage in Training

When building training datasets by simple SQL joins, it's easy to accidentally include feature values that were updated after the label event. The model trains on future data, gets great metrics, but fails in production because that data doesn't exist yet at prediction time.

Problem 4 — Low-Latency Serving is Hard

Features computed offline (batch) need to be available in milliseconds at serving time. Building and maintaining a pipeline from offline computation → online cache → API is significant infrastructure work.

Core Insight
Feast is a centralized contract between the offline world (training) and the online world (serving). Define a feature once, use it everywhere, safely.

2 The Feast Philosophy

A feature store is not a database. It is a time-aware, entity-centric view of your data — the same view at training time and serving time.

Three Core Beliefs

⏱️
Time is First-Class
Every feature value has a timestamp. Feast never gives you a feature value that didn't exist at the moment of prediction. This is the foundation of leakage prevention.
🧩
Define Once, Use Everywhere
Features are defined in one place. Training pipelines and serving systems both read from the same definition. No drift, no divergence.
🚀
Serve at Any Latency
Batch (offline) features for training. Materialized (online) features for millisecond serving. The same feature, two access patterns.

What Feast Is — and Is Not

Feast IS Feast is NOT
A registry of feature definitions A feature computation engine
A point-in-time correct retrieval layer A replacement for your data warehouse
A bridge between offline and online stores A streaming pipeline (by default)
A contract for training/serving parity An ETL tool
Common Misconception
Feast does not compute features. You compute features. Feast stores, versions, and serves them. The computation (pandas, Spark, SQL) happens upstream — Feast takes the output.

4 Core Concepts

Entity

The thing your features describe. The primary key of your feature store. In this demo it's a customer.

customer = Entity(
    name="customer_id",
    join_keys=["customer_id"],   # the column used to join
    value_type=ValueType.STRING,
)
Think of it this way
Entity is the "who" — customer, product, driver, session. Every feature you store belongs to an entity.

FileSource (Data Source)

Tells Feast where the raw feature data lives and which column is the event timestamp.

source = FileSource(
    path="data/telco_features.parquet",
    timestamp_field="event_timestamp",   # when was this feature value observed?
    created_timestamp_column="created_at",  # when was it ingested? (used for dedup)
)

created_timestamp_column is important when you have late-arriving data or backfills — Feast uses it to pick the most recently ingested value if multiple records exist for the same timestamp.

FeatureView

The central concept. It binds an entity + a data source + a schema together, with a TTL (how long a feature value is considered valid).

customer_stats = FeatureView(
    name="customer_stats",
    entities=[customer],
    ttl=timedelta(days=365),   # feature expires after 365 days
    schema=[
        Field(name="tenure",         dtype=Float32),
        Field(name="MonthlyCharges", dtype=Float32),
        # ... more fields
    ],
    source=source,
    online=True,    # allow materialization to online store
)

FeatureService

A named group of features for a specific use case — like a "serving contract." Lets you version what features a model uses.

svc = FeatureService(
    name="customer_stats_service",
    features=[customer_stats]   # reference the whole view, or specific fields
)

feast apply

Registers all definitions into the registry. Think of it like a database migration — it creates the schema without moving data.

store = FeatureStore(repo_path=str(REPO_DIR))
store.apply([customer, customer_stats])

TTL (Time-to-Live)

How stale a feature value can be before Feast ignores it. If a customer has no feature record within the TTL window before the query timestamp, Feast returns null rather than a misleadingly old value.

TTL ValueMeaning
timedelta(days=365)Accept feature values up to 1 year old
timedelta(days=1)Strict — only today's values count
timedelta(0)No limit — always return the latest value

1
Bootstrap the Feast Repo
Create directory structure, write feature_store.yaml. Wipe and recreate for a clean slate each run.
2
Load & Clean Data
Load Telco CSV, fix column names, coerce numeric types, drop nulls, create binary label from "Churn" column.
3
Simulate Timestamps (Demo-only)
Static CSV has no timestamps. Simulate feature_ts (when features were observed), label_ts (30 days later — when churn is known), created_at (ingestion lag). In production, real timestamps exist.
4
Encode Categoricals for Feast Storage
Feast's file/SQLite stores work with numeric types. LabelEncode all categorical columns. Note: in production you'd store raw + encode at model pipeline time.
5
Save to Parquet
Two parquet files: telco_features.parquet (feature values + event_timestamp) and entity_labels.parquet (entity IDs + label timestamps + labels). These are the inputs Feast reads from.
6
Define & Apply Feast Objects
Define Entity, FileSource, FeatureView. Run store.apply() to register them in the registry. No data is moved yet — only metadata.
7
get_historical_features() — Point-in-Time Join
Pass entity DataFrame with label_ts as the join timestamp. Feast retrieves only feature values that existed before each label timestamp. This is the leakage-safe training set.
8
Time-Based Split + Train
Split by timestamp quantile (70/15/15 train/val/test). Train logistic regression with sklearn pipeline. Tune classification threshold on validation set.
9
Materialize to Online Store
store.materialize_incremental() reads from offline parquet, writes latest feature values per entity into SQLite online store. Only the most recent value per customer is kept.
10
get_online_features() — Serving
Look up features by customer ID in milliseconds. Simulates what a production prediction API does. Returns the materialized feature row for each entity.

6 Point-in-Time Joins & Leakage Prevention

This is the most important technical concept in Feast. Understand this and everything else follows.

The Leakage Problem Visualized

Customer A timeline: Jan 1 Jan 15 Feb 1 │ │ │ ▼ ▼ ▼ [feature recorded] [customer churns] [support ticket filed] ✗ NAIVE JOIN (leaky): SELECT * FROM features JOIN labels ON customer_id → Feb 1 support ticket appears as a feature for a Jan 1 prediction → Model trains on data it could never have in production → Inflated accuracy, fails silently in prod ✓ POINT-IN-TIME JOIN (Feast): For label_ts = Feb 1 → only use features with event_timestamp ≤ Feb 1 AND event_timestamp > (Feb 1 - TTL) → Jan 1 features are used. Feb 1 ticket is excluded. → Perfectly reproduces what the model would see in production.

How get_historical_features() Works

entity_df = pd.read_parquet(entity_label_path)
# entity_df has: customer_id | event_timestamp (= label_ts) | label

training_df = store.get_historical_features(
    entity_df=entity_df,
    features=[f"customer_stats:{f}" for f in all_features],
).to_df()

For each row in entity_df, Feast finds the most recent feature record in the FeatureView where:

feature_ts ≤ entity_event_timestamp  AND  feature_ts ≥ (entity_event_timestamp - TTL)

Why Time-Based Split Matters Too

After getting the training set, we split by time — not random shuffle:

ts = training_df["event_timestamp"]
q_train = ts.quantile(0.70)   # first 70% of time → training
q_val   = ts.quantile(0.85)   # next 15% → validation
                                  # last 15% → test
Never Random-Split Time Series Data
A random split on temporal data lets the model "see" future patterns during training. Split by time so training always precedes validation which always precedes test — just like in production.

7 Offline vs Online: Two Access Patterns

Offline Store Online Store
Purpose Training dataset generation Real-time prediction serving
Latency Seconds to minutes Milliseconds
Data volume Full history Latest value per entity only
API get_historical_features() get_online_features()
Backend (demo) Local parquet files SQLite
Backend (prod) BigQuery, Redshift, Snowflake Redis, DynamoDB, Bigtable

Materialization — Bridging Offline to Online

Materialization reads from the offline store and writes the latest value per entity into the online store. Run it on a schedule to keep the online store fresh.

# Push all new feature values up to now into the online store
store.materialize_incremental(end_date=datetime.now(timezone.utc))

# Then serve in milliseconds
online = store.get_online_features(
    features=svc,
    entity_rows=[{"customer_id": "7590-VHVEG"}],
).to_dict()
Production Pattern
Run materialize_incremental() as a scheduled job (hourly/daily). Your prediction API then calls get_online_features() synchronously — no computation, just a key-value lookup.

8 Mental Model Summary

Think of Feast as Three Things at Once

📚
A Catalog
The registry is a catalog of "what features exist, what type they are, where they come from, and how long they're valid." Like a schema for your features.
🔎
A Time Machine
When building training data, Feast travels back to any point in time and reconstructs exactly what features were available then. No leakage, no hindsight.
🔌
A Cache Manager
Materialization copies offline features into a fast online cache. Your serving layer never re-computes — it just looks up precomputed values by entity ID.

The 5 Things to Always Remember

1. Entity      →  the "who" (customer_id, product_id, driver_id)
2. FeatureView →  the "what" (features for an entity + their source + TTL)
3. feast apply →  registers definitions in registry (no data moved)
4. get_historical_features() →  point-in-time safe training set
5. materialize + get_online_features() →  production serving

The Golden Rule

Your model in production sees exactly the same feature values it trained on — because both are served by the same Feast definitions. That is training/serving parity. That is why Feast exists.
When to use Feast in a real project
When you have: multiple models sharing features · a gap between training and serving environments · requirements for reproducible training sets · millisecond serving latency constraints · regulatory need to audit what data a model saw at training time.
