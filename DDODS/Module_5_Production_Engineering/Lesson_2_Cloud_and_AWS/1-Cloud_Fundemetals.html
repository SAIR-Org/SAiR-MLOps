Cloud Fundamentals

An overview of cloud concepts that matter: virtualization, storage choices, VPC, load balancing, identity, and observability.

Basics: how the internet works

Every cloud solution is built on fundamental networking principles. Let's discuss the most fundamental ones, briefly and abstractly:

IP addresses and domain names

Every device connected to the internet requires an IP address. We primarily deal with two types: IPv4 and IPv6. IPv6 was created because IPv4 addresses were running out as the Internet grew. Since humans are not good at remembering long numbers, we use domain names instead. This is where DNS (Domain Name System) comes in, which converts user-friendly domain names into the numerical IP addresses that computers understand.

Data movement

When you load a website or send an email, data doesn't move as one big chunk. Your information is broken down into small pieces called packets. Each packet carries the data itself, along with crucial information like the destination IP and the source IP. The system managing this transfer is TCP/IP. TCP handles breaking down the data and ensuring it arrives correctly, while IP ensures the packets reach the right destination.

Cloud computing: definition and characteristics

Cloud computing refers to delivering computing services (e.g., servers, storage, databases, networking, software, analytics) over the internet ("the cloud"), enabling users and organizations to access them on demand. Rather than owning physical hardware, organizations can rent virtualized resources, scale them elastically, and pay only for what they use.

NIST essential characteristics

According to NIST (National Institute of Standards and Technology), any cloud system should exhibit the following five essential characteristics:

  • On-demand self-service — Users can provision computing capabilities on their own without requiring interaction with the provider's staff.
  • Broad network access — Services are accessible over the network via standard mechanisms from heterogeneous client platforms.
  • Resource pooling / multi-tenancy — The provider's resources are pooled to serve multiple consumers, dynamically assigning resources based on demand.
  • Rapid elasticity / scalability — Capabilities can be elastically scaled up or down to meet fluctuating demands.
  • Measured service (metering) — Resource usage is monitored, controlled, and reported, enabling pay-per-use models.
These characteristics help distinguish cloud computing from traditional hosting or fixed-capacity infrastructure.

Cloud economics & cost models

  • Pay-as-you-go — pay for actual usage (compute hours, storage size, network in/out).
  • Reserved/committed capacity — discounts for committing to use resources over a period.
  • Spot/preemptible instances — very low-cost compute nodes that can be reclaimed (ideal for training or batch workloads).
  • Cost leakage & overprovisioning — without good governance, unused resources can accumulate costs.
A good mental model: cloud resources = utility resources (like electricity). You want to scale dynamically and only pay for what you use.

Shared responsibility model

In the cloud, security and operational responsibilities are shared between the provider and the user. The provider ensures physical infrastructure, network, host OS, and hypervisor security. The user handles secure configuration of the OS, network rules, application logic, data encryption, IAM, and compliance.

Virtual Machines, hypervisors, and virtualization

At the core of modern cloud computing lies virtualization — the process of creating a virtual version of something, such as a server, storage device, or network resource. Virtualization enables multiple isolated computing environments to coexist on a single physical machine.

Hypervisor

A hypervisor, also known as a Virtual Machine Monitor (VMM), is the software layer that sits between the physical hardware and virtual machines. Its primary role is to abstract the physical hardware and allocate computing resources to multiple virtual machines (VMs).

Type 1 — Bare-metal
Run directly on the host's hardware. Examples: VMware ESXi, Microsoft Hyper-V, KVM, Xen.
Type 2 — Hosted
Run on top of a host operating system. Examples: Oracle VirtualBox, VMware Workstation, Parallels Desktop.

How virtual machines work

  • Virtual CPU (vCPU) — emulates physical CPU cycles allocated from the host.
  • Virtual Memory — allocated from the host's RAM.
  • Virtual Storage — uses disk images to emulate storage.
  • Virtual Network Interface — connects to the virtual or physical network.

Each VM runs its own full operating system (guest OS), which is independent from others. This allows for strong isolation between VMs and the ability to run different OS types on the same hardware.

Benefits of VMs

  • Isolation — each VM is sandboxed; failures don't affect others.
  • Resource allocation — physical resources are efficiently shared and allocated dynamically.
  • VM migration — VMs can be moved between physical hosts with minimal downtime.
  • Snapshot & cloning — easy to create snapshots or clone VMs for backup and scalability.
  • Multi-tenancy — enables multiple users or businesses to share the same physical infrastructure securely.

Limitations of VMs

  • Overhead — each VM runs a full OS, consuming CPU, memory, and storage.
  • Boot time — VMs typically take longer to boot compared to containers.
  • Resource waste — unused allocated resources still count as in-use, reducing efficiency.
  • Limited scalability speed — spinning up or scaling VMs is slower than container-based deployments.

Containers & orchestration (Kubernetes)

  • Container — a lightweight packaging of the application + dependencies, running isolated but sharing the host OS kernel.
  • Kubernetes — an orchestration layer that schedules containers (pods), handles scaling, rolling updates, service discovery, and more.
  • Managed container services — cloud providers offer managed Kubernetes services: Amazon EKS, Google GKE, Azure AKS. These offer reliability, control-plane maintenance, autoscaling, and integration with cloud tooling.

Storage systems: Block, Object, File

Object Storage

Cloud object storage (e.g., Amazon S3, Azure Blob, GCS) is built for handling massive amounts of unstructured data. It organizes data as objects, not files within folders. Each object contains three key components: the data itself, metadata, and a unique identifier. There's no concept of nested directories — everything exists in a flat namespace. Access typically happens over HTTP-based APIs.

Best for: data lakes, media repositories, log archives, backups, static website hosting. Trade-offs: not for transactional workloads, higher latency than block storage, no in-place modification.

Block Storage

Block storage (e.g., Amazon EBS, Azure Disk Storage) provides the foundation for high-performance, low-latency workloads. It works by storing data in fixed-size blocks, forming virtual disks that can be mounted to your operating system. Once mounted, the OS formats the block device and manages its own file system.

Best for: databases (MySQL, PostgreSQL), OS boot disks, transactional systems. Trade-offs: limited capacity per volume, tied to a single compute instance, more expensive per GB than object storage.

File Storage / Shared File Systems

File storage (e.g., Amazon EFS, Azure Files, NFS, SMB) provides a familiar way to store and access data: as files organized in folders. It uses a hierarchical structure of directories and files, supporting traditional file system semantics such as permissions, locks, and shared access.

Best for: shared project directories, enterprise home folders, content management systems, ML pipelines with shared datasets. Trade-offs: performance varies with network throughput, more expensive and less scalable than object storage.

Databases and warehouses

Databases and data warehouses (e.g., MySQL, PostgreSQL, MongoDB, Snowflake, BigQuery, Redshift) sit one layer above raw storage systems. They don't just store data but rather give it structure, meaning, and accessibility. They provide query languages, transactions, indexing, concurrency control, and schema enforcement.

Object Storage
Most scalable and cost-effective for unstructured or archival data.
Block Storage
Best for performance-sensitive workloads requiring low latency and frequent updates.
File Storage
Useful when multiple systems need to share files with traditional file system semantics.
Databases & Warehouses
Higher abstraction layer for structured data and complex queries.

Networking in the cloud

Cloud networking forms the backbone of all communication between compute, storage, and external users. Unlike traditional on-premise data centers, cloud networks are software-defined, meaning every aspect of connectivity can be configured programmatically.

VPC (Virtual Private Cloud) / Virtual Network

A VPC is essentially your private, customizable network inside a cloud provider's infrastructure. It lets you carve out a dedicated portion of the provider's massive network and define how your resources connect within it. Inside a VPC, you can define your IP address range, create subnets to organize and isolate workloads, and control connectivity.

Main goal: isolation and control. Ensure workloads are separated from others, while still giving full control over traffic flow and security.

Subnets and availability zones

Subnets divide your VPC into smaller, manageable sections with their own routing rules and accessibility level. Cloud providers divide each region into multiple Availability Zones (AZs) — physically separate data centers that operate independently. By distributing workloads across multiple AZs, you eliminate single points of failure.

Load Balancers and API Gateways

  • Load Balancer (LB) — distributes incoming requests across multiple compute instances or containers, ensuring no single resource becomes a bottleneck. Adds fault tolerance and boosts performance.
  • API Gateway — sits at the entry point of your architecture. Routes client requests, manages authentication, enforces rate limits, and performs caching or request transformation.

Networking architecture example for ML deployment

1
Users / Clients — send inference requests to your model endpoint.
2
API Gateway — handles authentication, routing, and rate limiting.
3
Load Balancer — distributes traffic across multiple inference servers.
4
Inference Servers — access model artifacts from object storage (S3, Blob Storage).
5
VPC — isolates the entire setup. Subnets separate public-facing components from private ones.

Identity & IAM

IAM is a framework for managing identities, roles, and permissions. It defines who can access which resources and what actions they're allowed to perform.

Users
Individual identities — people or systems that need to interact with cloud resources.
Groups
Collections of users with similar responsibilities — manage permissions collectively.
Roles
Define what actions can be performed — can be temporarily assumed by users, applications, or services.
Policies
Actual rules governing access — JSON documents specifying what's allowed or denied on specific resources.

Service roles and roles for Pods

  • Service Roles — cloud services like EC2, Lambda, or Vertex AI can be assigned roles directly, allowing them to access resources like databases or secret managers without embedding credentials in code.
  • Roles for Pods (EKS) — individual pods can assume specific IAM roles that grant only the permissions they need. Provides fine-grained access control, credential isolation, and reduced risk exposure.

Security groups / Firewall rules

Security groups act as virtual firewalls attached to your cloud resources, controlling what traffic gets in and what goes out. Inbound rules determine which external requests can reach your instance, while outbound rules define which destinations your resources can connect to.

Principle of least privilege: every identity should have only the permissions it absolutely needs and nothing more. This minimizes the attack surface and makes compliance audits straightforward.

Monitoring, logging, observability

Observability is the backbone of trustworthy production systems, providing the necessary insight to maintain reliability and performance. It encompasses the collection and analysis of metrics and telemetry data, such as CPU usage, memory consumption, disk I/O, and network activity.

  • Metrics — CPU, memory, disk I/O, network activity, GPU usage.
  • Logs — application logs, system logs, audit logs.
  • Traces — end-to-end request tracing for debugging latency issues.

Centralized tools like Prometheus, Grafana, and CloudWatch play a crucial role in aggregating, visualizing, and managing this data effectively.

Patterns for ML workloads in Cloud

Training vs Inference vs Batch

Training
Compute-hungry, often distributed across GPUs/TPUs. Heavy I/O for large datasets. Runs for hours or days.
Real-time inference
Low latency, high concurrency, predictable tail performance. Autoscaling and model versioning critical.
Batch inference
Throughput matters, speed per request doesn't. Runs on ephemeral clusters or spot instances.

Data pipelines & storage

  • Raw data — ingested into staging area (S3, Blob Storage).
  • Validation & transformation — deduplication, quality checks, feature engineering.
  • Storage — raw/processed data in object storage; high-throughput training datasets in block storage; feature stores for low-latency lookups.

Model Registry & versioning

  • Model Registry — single source of truth for model artifacts. Tracks weights, hyperparameters, environment details, dataset versions.
  • Lifecycle — registering, staging, promoting to production, rolling back.
  • Auditability — record training code, library versions, serialized formats, input/output schemas.

Inference serving patterns

Serverless inference
Ideal for small models or unpredictable workloads. Autoscales automatically. Trade-off: cold starts, limited memory.
Container-based microservices
Go-to for production-grade deployments. Full control over runtime. Managed via Kubernetes and model servers like TorchServe or Triton.
Edge inference
For ultra-low-latency use cases. Models quantized, pruned, or converted to ONNX, deployed to edge devices.
Batch scoring
Periodic analytics or re-scoring historical data. Runs on distributed compute frameworks like Spark or Ray.

Autoscaling

  • Horizontal Pod Autoscaling (HPA) — adjusts replica counts based on CPU, memory, or custom metrics.
  • GPU autoscaling — combines cluster autoscalers with GPU-aware schedulers.
  • Predictive scaling — for predictable traffic surges, maintain warm pools or preemptively spin up replicas.

Deployment strategies

Blue-Green
Deploy new version to parallel environment; switch traffic after validation. Zero-downtime.
Canary Deployments
Route small percentage of traffic to new model; monitor metrics before wider rollout.
A/B Testing
Experiment with different models for feature/metric comparison.
Shadow Deployment
New model receives production traffic but responses are for internal evaluation only.

Monitoring and Detection

  • Model quality — accuracy, AUC, precision/recall, task-specific metrics.
  • Data & prediction drift — input feature distributions, population shifts, prediction distribution changes.
  • Resource & performance — latency percentiles, throughput, GPU/CPU/memory, disk usage and error rates.

Reproducibility & auditability

  • Traceability — record and link training dataset versions, preprocessing code, hyperparameters, random seeds, evaluation metrics.
  • Reproducibility — capture runtime environments (Docker images, conda/pip lockfiles), snapshot datasets or store dataset hashes.
  • Auditability — capture access logs and model actions for audits, ensure sensitive data handling complies with regulations.

Conclusion

  • Cloud computing forms the foundation for scalable and efficient ML deployment through flexible deployment, service, and responsibility models.
  • A clear understanding of cloud infrastructure, networking, and IAM is crucial for building secure and well-managed ML pipelines.
  • Load balancing and scalability are key players in maintaining performance and reliability by distributing workload evenly across multiple servers.
  • Storage choices (object, block, file, databases) must match workload characteristics — throughput, access patterns, concurrency, and cost.
  • Observability — metrics, logs, and traces — is essential for maintaining production-grade ML systems.
  • Reproducibility and auditability require rigorous tracking of datasets, code, environments, and deployment history.
