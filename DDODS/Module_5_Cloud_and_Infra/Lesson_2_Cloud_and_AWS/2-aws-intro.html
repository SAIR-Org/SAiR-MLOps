What is AWS?

Amazon Web Services is a comprehensive cloud computing platform providing IT resources — compute, storage, databases, and 200+ services — over the internet. It lets businesses run applications without managing physical servers or data centers.

EKS — Elastic Kubernetes Service

When Kubernetes became the de facto standard for container orchestration, it also brought significant operational complexity — cluster setup, upgrades, networking, HA. EKS was built to remove that undifferentiated heavy lifting.

EKS in one sentence: Amazon's managed Kubernetes control plane — delivers upstream-compliant Kubernetes with integrated AWS identity, networking, and scaling, while AWS operates, patches, and scales the control plane components.

The two planes

Control plane — managed by AWS
API server, etcd, controller manager, scheduler
Automatically provisioned across 3 AZs for fault tolerance. You interact only via the Kubernetes API endpoint. AWS handles patching, upgrades, and etcd backups. Scales automatically and is fully redundant.
Data plane — managed by you
Worker nodes — EC2 or Fargate
The actual compute where pods run. You choose: EC2 instances for full control over instance type and scaling, or Fargate (serverless) where AWS provisions compute per pod — no node management at all.

Data plane options compared

EKS on EC2
Manage EC2 node groups. Full control over instance type, scaling, AMI. Uses the EKS-optimized AMI preconfigured with kubelet, containerd, and VPC CNI. Ideal for performance-sensitive workloads.
EKS on Fargate
AWS provisions compute automatically per pod. Define Fargate profiles mapping namespaces and labels. No node management. Limits some customization. Best for event-driven or short-lived tasks.

Security model

EKS uses a dual-layer security model: IAM for cluster-level access control, and standard Kubernetes RBAC for in-cluster permissions. The API server endpoint can be public (IAM-secured) or private (VPC-only). etcd is encrypted at rest.

EC2 & node provisioning

Amazon EC2 (Elastic Compute Cloud) provides the raw compute foundation. It's a virtualization service offering resizable compute capacity — you choose instance type (CPU, memory, GPU), storage (EBS), and networking (VPC). EKS builds its worker layer directly on EC2.

Node provisioning options

✓ Managed node groups — recommended
AWS automatically provisions and manages the EC2 lifecycle. You define instance type, desired capacity, and scaling config. AWS creates the Auto Scaling group, joins instances to the cluster, and handles rolling upgrades with minimal downtime.
Self-managed nodes
You provision EC2 instances manually (CloudFormation, Terraform) and join them via aws-auth ConfigMap. Maximum control, but full operational responsibility for scaling, health checks, and updates.

Node lifecycle & scaling

  • Each EC2 worker node runs the kubelet agent — it registers the node, reports status, and ensures pods are running
  • The EKS-optimized AMI includes all required configuration; nodes auto-register when launched with the correct bootstrap parameters
  • The Cluster Autoscaler integrates with EC2 Auto Scaling groups — adds nodes when pods are unschedulable, removes them when underutilized
  • Combined with EC2's pay-as-you-go model, this creates a fully elastic compute environment

Networking & security

  • The AWS VPC CNI plugin assigns IP addresses to pods directly from the VPC's CIDR range — pods are native VPC endpoints
  • No NAT required: traffic stays inside the VPC, making pod-to-AWS-resource communication more secure and efficient
  • Security Groups can be applied per node; network policies restrict pod-to-pod traffic
  • IAM roles on EC2 instances let workloads assume fine-grained permissions to access other AWS resources

EKS networking & add-on services

Running Kubernetes in production requires more than control and data planes. AWS provides managed add-ons that tie native AWS services into your cluster — configurable automatically through eksctl.

Service / add-on Type What it does
AWS VPC CNI Default Assigns native VPC IPs to pods; removes need for overlay networks; runs as a DaemonSet
CoreDNS Default Internal cluster DNS — workloads address each other by service name, not IP; 2 replicas by default
kube-proxy Default Network routing rules per node; bridges Kubernetes Services and pods for NodePort/LoadBalancer traffic
Amazon ECR Default Native container registry; auto-integrates with EKS managed node groups — no extra setup required
Amazon EBS (root) Default Root storage for every EC2 worker node; provisioned and deleted automatically with node lifecycle
Cluster Autoscaler Optional Monitors unschedulable pods; adds/removes EC2 nodes in Auto Scaling groups autonomously with proper IAM
AWS Load Balancer Controller Optional Enables ALB/NLB, Ingress management, path-based routing, SSL termination (extends built-in CLB support)
Amazon EBS CSI driver Optional Persistent block storage volumes for stateful workloads like databases and analytics pipelines
CloudWatch + Fluent Bit Optional Fluent Bit DaemonSet ships container logs to CloudWatch; metrics/traces feed dashboards — no external monitoring needed
AWS Fargate Optional Serverless data plane; each pod in its own isolated runtime; no EC2/EBS to manage; suits stateless, short-lived workloads

Strategic implications for organizations

  • Platform engineering mindset: Manage multiple AWS accounts centrally. Build internal service catalogs, standardized landing zones, CI/CD pipelines, and cost-security guardrails — so teams consume pre-built, compliant capabilities efficiently.
  • Cost governance from day one: Managed services can scale and proliferate rapidly, making costs easy to miss. Embed cost-tracking, budgeting, and resource tagging from the outset to prevent surprise bills.
  • Security as a first-class citizen: Implement least-privilege IAM, strong network isolation, encryption in transit and at rest, and continuous audit logging. Integrate these early to avoid friction later in the development lifecycle.
  • Deep observability: Traditional CPU and memory metrics are not enough. Monitor service interactions, latency, data transfers, and cost anomalies. AWS-native CloudWatch provides these insights without external stacks.
  • Manage vendor lock-in risk: Deep AWS-native adoption brings efficiency but limits long-term flexibility. Maintain abstraction layers and consider multi-cloud setups where practical to preserve future options.
  • Continuous learning is non-negotiable: AWS evolves rapidly with new services and best practices. Teams must invest in ongoing certifications, architectural reviews, and training to keep systems efficient and aligned with goals.
Key takeaway: EKS combines default add-ons (VPC CNI, CoreDNS, kube-proxy, ECR, EBS) with optional integrations (Cluster Autoscaler, Load Balancer Controller, EBS CSI, CloudWatch) to deliver a production-ready environment — transforming Kubernetes from a standalone orchestration engine into a unified cloud operations platform.
